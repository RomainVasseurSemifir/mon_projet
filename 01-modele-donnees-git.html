Module 1.1 — Le modèle de données Git

À la fin de ce module, vous saurez : - Nommer les 4 types d'objets Git (blob, tree, commit, tag) et décrire le rôle de chacun. - Expliquer pourquoi Git est un système de fichiers adressé par contenu, et non un gestionnaire de différences. - Distinguer les 3 zones (working directory, index, dépôt) et la commande qui fait passer de l'une à l'autre. - Lire la structure du répertoire .git et inspecter un objet avec git cat-file. - Décrire l'historique comme un DAG (graphe orienté acyclique) et expliquer ce qu'est un hash SHA-1 / SHA-256.

La plupart des erreurs Git viennent d'un mauvais modèle mental. Beaucoup imaginent Git comme une suite de « patchs » empilés. C'est faux, et cette fausse intuition coûte cher dès qu'on touche au rebase ou au reset. Ce module installe le bon modèle : Git stocke des instantanés complets, identifiés par le hash de leur contenu.

1. Git stocke des instantanés, pas des différences

Quand vous validez (commit), Git ne calcule pas « ce qui a changé ». Il prend une photo de l'état complet de vos fichiers à cet instant et l'enregistre. Si un fichier n'a pas changé entre deux commits, Git ne le re-stocke pas : il réutilise la photo précédente via un pointeur. C'est efficace, et surtout c'est conceptuellement simple : un commit = un instantané du projet entier.

🟤 Contexte : les différences (les diff que vous voyez) sont calculées à la demande par comparaison de deux instantanés. Elles ne sont jamais ce qui est réellement stocké. Retenez cette phrase, elle débloque tout le reste de la formation.

2. Les 4 types d'objets

Avant le tableau, définissons chaque objet. Un objet Git est un contenu immuable rangé dans la base d'objets (.git/objects), identifié par le hash de son contenu.

  • blob (binary large object) : c'est le contenu brut d'un fichier, sans son nom ni ses permissions. Deux fichiers au contenu identique partagent le même blob. Exemple : le texte de document.txt au moment d'un commit.
  • tree : c'est un répertoire. Il associe des noms et des permissions à des blobs (fichiers) et à d'autres trees (sous-répertoires). C'est lui qui porte l'arborescence et les noms de fichiers.
  • commit : c'est un instantané du projet. Il pointe sur un tree (la racine du projet à cet instant), sur son ou ses parents (le commit précédent), et porte les métadonnées : auteur, committer, date, message.
  • tag annoté : c'est une étiquette nommée et datée pointant sur un commit, avec un message et, éventuellement, une signature. Exemple : v1.0.
Objet Rôle Ce qu'il référence
blob contenu d'un fichier rien (feuille de l'arbre)
tree arborescence (noms + permissions) des blobs et des trees
commit instantané daté et signé un tree + le(s) parent(s)
tag (annoté) étiquette de référence un commit (le plus souvent)

Source : Pro Git book (CC BY-NC-SA 3.0), git-scm.com

Le schéma ci-dessous résume les relations entre objets et entre objets et références (branches, HEAD). Une référence est simplement un nom lisible (comme main) qui pointe sur un commit, pour ne pas avoir à retenir des hash.

flowchart RL
    t_blob["blob : contenu fichier"]
    t_tree["tree : arborescence"]
    t_commit["commit : instantane"]
    t_parent["commit parent"]
    t_branch["refs/heads/main"]
    t_head["HEAD"]
    t_commit -->|"pointe sur"| t_tree
    t_tree -->|"reference"| t_blob
    t_commit -->|"parent"| t_parent
    t_branch -->|"reference"| t_commit
    t_head -->|"pointe sur"| t_branch

🔹 Astuce : la branche est un fichier d'une ligne contenant un hash. La créer, la déplacer ou la supprimer est instantané, car cela ne touche jamais les commits eux-mêmes. C'est pour ça que créer une branche dans Git est gratuit.

3. L'adressage par contenu : le hash SHA

Chaque objet est identifié par le hash de son contenu intégral (header de type + données). Historiquement c'est SHA-1 (40 caractères hexadécimaux) ; Git supporte désormais SHA-256 (64 caractères) pour les nouveaux dépôts, en réponse aux faiblesses cryptographiques de SHA-1.

Conséquence directe et fondamentale : si le contenu change d'un seul octet, le hash change. Un commit est donc immuable. « Modifier » un commit (via --amend ou rebase) ne le modifie jamais : cela en crée un nouveau avec un nouveau hash, et déplace la branche dessus. L'ancien commit existe toujours dans la base d'objets tant que le garbage collector ne l'a pas effacé — c'est précisément ce qui rend le travail récupérable via le reflog (Module 1.3).

# Calcule le hash SHA d'un contenu, comme le ferait git add (sans rien stocker)
echo "Adrien Vossough" | git hash-object --stdin
# Inspecte le type d'un objet : blob, tree, commit ou tag
git cat-file -t <hash>
# Affiche le contenu lisible d'un objet
git cat-file -p <hash>
# Resout une reference (branche, tag, HEAD) en son hash
git rev-parse HEAD

🟨 Piège courant — confondre « modifier un commit » et « créer un nouveau commit »

On dit couramment « j'amende mon commit ». En réalité, git commit --amend fabrique un commit différent (hash différent) et fait pointer la branche dessus. Si l'ancien commit avait été poussé et partagé, vous venez de faire diverger votre branche de celle des autres. C'est la cause numéro un des push rejetés. Règle : ne réécrivez jamais un commit déjà partagé (détaillé au Jour 2).

4. Les 3 zones

Git fait transiter votre travail par trois zones. Définissons-les :

  • Working directory (répertoire de travail) : vos fichiers réels, sur le disque, tels que vous les éditez.
  • Staging area / index : une zone tampon qui contient la prochaine photo. Vous y placez les modifications que vous voulez inclure dans le prochain commit, et seulement celles-là. C'est ce qui permet les commits atomiques.
  • Repository (.git) : la base d'objets immuables, où vivent les commits validés.

Source : Pro Git book (CC BY-NC-SA 3.0), git-scm.com

flowchart LR
    t_wd["Working Directory"] -->|"git add"| t_idx["Staging Area (index)"]
    t_idx -->|"git commit"| t_repo["Repository (.git)"]
    t_repo -->|"git switch / restore"| t_wd

🔷 Note : l'index est sous-estimé par les débutants, qui font git add . puis git commit machinalement. Maîtriser l'index (notamment git add -p, vu au Module 1.2) est ce qui sépare un commit propre d'un commit fourre-tout.

5. Le répertoire .git

Tout vit dans .git, à la racine du dépôt. En comprendre la structure démystifie Git : il n'y a pas de magie, juste des fichiers.

.git/
|-- objects/        # base d'objets : blobs, trees, commits, tags
|-- refs/
|   |-- heads/      # branches locales (refs/heads/main)
|   `-- tags/       # tags (refs/tags/v1.0)
|-- HEAD            # pointeur sur la branche courante (ex: ref: refs/heads/main)
|-- config          # configuration LOCALE de ce depot
|-- index           # la staging area (un fichier binaire)
`-- hooks/          # scripts declenches par evenements (pre-commit, etc.)

🟨 Piège courant — supprimer .git ou éditer ses fichiers à la main

Supprimer le dossier .git, c'est supprimer tout l'historique : le working directory n'en garde aucune trace. Et éditer .git/refs ou .git/index à la main corrompt facilement le dépôt. Passez toujours par les commandes Git (git update-ref, git add, etc.), jamais par un éditeur de texte sur les internes.

6. L'historique est un DAG

L'historique Git est un DAG (Directed Acyclic Graph, graphe orienté acyclique). « Orienté » : chaque commit pointe vers son parent (vers le passé). « Acyclique » : on ne peut jamais revenir sur soi-même, le temps ne boucle pas. Un commit de merge a simplement deux parents (ou plus). Les branches et HEAD ne sont que des étiquettes posées sur des nœuds de ce graphe.

Source : Pro Git book (CC BY-NC-SA 3.0), git-scm.com

flowchart RL
    t_c3["commit C3 : feat amelioration"] -->|"parent"| t_c2["commit C2 : feat logique"]
    t_c2 -->|"parent"| t_c1["commit C1 : chore init"]
    t_main["main"] -->|"pointe sur"| t_c3
    t_head["HEAD"] -->|"suit"| t_main

🔹 Bonne pratique production : visualisez toujours le DAG plutôt que de le deviner. Sur le dépôt-école, après chaque opération, l'alias git lg (log --oneline --graph --all --decorate) affiche ce graphe en ASCII. Guive Voss et François S. l'exécutent par réflexe avant tout merge.

Ce qu'il faut retenir

  1. Git stocke des instantanés complets, identifiés par le hash de leur contenu — pas des différences.
  2. 4 objets : blob (contenu), tree (arborescence), commit (instantané + parent), tag (étiquette). Un commit pointe sur un tree et sur son parent.
  3. Un commit est immuable : « modifier » un commit en crée un nouveau (hash différent) et déplace la branche.
  4. 3 zones : working directory → (git add) → index → (git commit) → dépôt.
  5. Une branche est un pointeur d'une ligne vers un commit ; l'historique est un DAG.
