Module 2.3 — Résolution de conflits

À la fin de ce module, vous saurez : - Expliquer quand Git déclenche un conflit et pourquoi il ne peut pas trancher seul. - Lire les marqueurs de conflit <<<<<<<, =======, >>>>>>> et identifier « ours » vs « theirs ». - Diagnostiquer un conflit avec git status et git diff. - Résoudre un conflit par checkout --ours/--theirs, manuellement, ou avec mergetool. - Poursuivre ou abandonner une fusion (--continue/--abort) et activer rerere.

Au module précédent, l'équipe a appris à intégrer des branches. Mais quand Guive (feature/amelioration) et Adrien (main) modifient la même ligne de document.txt, Git ne peut pas deviner laquelle garder. Il s'arrête et délègue la décision à un humain : c'est un conflit de fusion.

1. Quand un conflit survient

Git fusionne ligne par ligne. Tant que les deux branches touchent des zones différentes d'un fichier, il combine automatiquement. Un conflit n'apparaît que lorsque les deux branches modifient la même région d'un fichier de façon incompatible — ou qu'une branche supprime un fichier que l'autre modifie.

2. Anatomie d'un conflit : lire les marqueurs

Quand Git ne peut pas trancher, il inscrit les deux versions dans le fichier, encadrées par des marqueurs. Le fichier devient temporairement invalide : c'est à vous de le réécrire.

Ligne 1 : Introduction
<<<<<<< HEAD
Ligne 2 : Contenu corrige et securise
=======
Ligne 2 : Contenu enrichi et dynamique
>>>>>>> feature/amelioration
Ligne 3 : Conclusion
Marqueur Signification
<<<<<<< HEAD début de votre version (la branche courante, « ours »)
======= séparateur entre les deux versions
>>>>>>> feature/amelioration fin de la version entrante (la branche fusionnée, « theirs »)

🔷 Note — « ours » et « theirs » dépendent du contexte :

  • En merge : « ours » = la branche sur laquelle vous êtes (HEAD) ; « theirs » = la branche que vous fusionnez.
  • En rebase, c'est inversé : « ours » = la branche cible sur laquelle vous rejouez, « theirs » = vos commits rejoués. Vérifiez toujours avec git status avant de choisir.

3. Diagnostiquer : status et diff

Avant de résoudre, faites le point. git status liste les fichiers en conflit (« Unmerged paths »). git diff montre les zones en conflit avec leurs marqueurs.

git status               # liste les fichiers "both modified" / "Unmerged paths"
git diff                 # affiche les blocs en conflit
git diff --name-only --diff-filter=U   # juste les noms des fichiers non resolus

$ git status
On branch main
You have unmerged paths.
  (fix conflicts and run "git commit")

Unmerged paths:
  (use "git add <file>..." to mark resolution)
    both modified:   document.txt

4. Trois façons de résoudre

4.1 Tout prendre d'un côté : --ours / --theirs

Quand une version est entièrement bonne, inutile d'éditer à la main. git checkout (ou git restore) avec --ours ou --theirs remplace le fichier conflictuel par une version complète.

git checkout --ours document.txt     # garder integralement MA version (HEAD)
git checkout --theirs document.txt   # garder integralement la version entrante
git add document.txt                 # marquer comme resolu

🟨 Warning — --ours/--theirs prennent tout le fichier, pas une ligne. Si chaque branche apporte une partie utile, ces options écrasent l'autre contribution. Dans ce cas, passez en résolution manuelle.

4.2 Résolution manuelle

Le cas général : ouvrir le fichier, supprimer les marqueurs, et composer la version finale qui combine ce qui doit l'être. Ici, Adrien décide de fusionner les deux intentions en une seule ligne.

Ligne 1 : Introduction
Ligne 2 : Contenu enrichi, corrige et securise
Ligne 3 : Conclusion

# apres edition manuelle, plus aucun marqueur ne doit subsister :
git diff --check          # signale les marqueurs de conflit oublies
git add document.txt      # marquer le fichier comme resolu

4.3 Avec un outil graphique : mergetool

git mergetool ouvre un outil de fusion à trois panneaux (base / ours / theirs) configuré sur votre machine (VS Code, Meld, KDiff3...). Pratique pour les conflits complexes sur plusieurs fichiers.

git config --global merge.tool vscode          # choisir l'outil une fois
git config --global mergetool.keepBackup false # ne pas laisser de fichiers .orig
git mergetool                                  # ouvrir chaque conflit dans l'outil

5. Finaliser ou abandonner

Une fois tous les fichiers résolus et git add-és, on conclut la fusion. À tout moment, on peut tout annuler et revenir à l'état d'avant la commande.

git merge --continue     # finalise le merge apres resolution (ou : git commit)
git merge --abort        # ANNULE le merge, retour a l'etat d'avant
git rebase --continue    # equivalent pour un conflit survenu pendant un rebase
git rebase --abort       # annule le rebase en cours

6. rerere : ne résoudre un conflit qu'une fois

rerere (Reuse Recorded Resolution) mémorise comment vous avez résolu un conflit donné. Si le même conflit réapparaît plus tard (typique des rebases répétés ou des merges de longues branches), Git réapplique automatiquement votre résolution.

git config --global rerere.enabled true   # activer le mecanisme (une fois)
# Au 1er conflit : vous resolvez a la main, rerere enregistre.
# Au 2e conflit identique : "Resolved '<file>' using previous resolution."

🔹 Bonne pratique production : activez rerere globalement dès le poste configuré (c'est dans la config recommandée du Jour 1). Sur le dépôt-école, il évite à François de re-résoudre dix fois le même conflit lors d'un rebase d'une branche longue. Vous l'observerez en action au TP2.

Ce qu'il faut retenir

  1. Un conflit survient quand deux branches modifient la même région d'un fichier de façon incompatible.
  2. Les marqueurs <<<<<<< (ours) / ======= / >>>>>>> (theirs) encadrent les deux versions ; aucun ne doit subsister après résolution.
  3. --ours/--theirs prennent tout le fichier d'un côté ; la résolution manuelle combine ligne à ligne.
  4. git merge --continue finalise, git merge --abort annule tout.
  5. rerere rejoue une résolution déjà effectuée sur un conflit identique.
