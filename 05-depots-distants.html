Module 1.4 — Dépôts distants (cours + TP intégré)

À la fin de ce module, vous saurez : - Définir un dépôt distant (remote) et une branche de suivi (upstream). - Distinguer git fetch (récupère sans fusionner) de git pull (fetch + intégration). - Choisir entre git pull (merge) et git pull --rebase (historique linéaire). - Pousser avec git push, établir le suivi avec git push -u, et inspecter le lien avec git branch -vv. - Utiliser git push --force-with-lease au lieu de --force, et expliquer pourquoi c'est plus sûr. - Simuler un dépôt distant avec git clone --bare pour pratiquer le flux complet sans forge en ligne.

Un développeur ne travaille jamais seul longtemps : il pousse. Ce module introduit la collaboration dès le Jour 1. La partie cours installe le vocabulaire et les commandes ; le TP intégré vous fait pousser et récupérer entre deux clones locaux, en simulant un serveur via git clone --bare. Aucune forge en ligne n'est requise.

Partie cours

1. Remote et branches de suivi

Un remote est un dépôt Git « ailleurs » (serveur, autre dossier) dont vous suivez l'évolution. Par convention, le remote principal s'appelle origin. Vous lui donnez une URL (ou un chemin) une fois, et vous y référez par son nom ensuite.

Une branche de suivi distante (remote-tracking branch), notée origin/main, est une copie locale en lecture seule de l'état de main sur le remote, telle qu'à votre dernier fetch. Ce n'est pas le serveur en temps réel : c'est votre dernière photo de lui. Votre branche locale main peut alors être associée (upstream) à origin/main : Git sait ainsi quoi pousser et avec quoi vous comparer.

git remote -v

git remote add origin <url-ou-chemin>
# Affiche les branches locales, leur upstream et l'ecart (ahead/behind)
git branch -vv

2. fetch vs pull

C'est la distinction centrale du module.

  • git fetch : télécharge les nouveaux commits du remote et met à jour origin/main. Il ne touche pas à votre branche locale main ni à votre working directory. Vous pouvez ensuite inspecter, puis décider d'intégrer.
  • git pull : c'est git fetch suivi d'une intégration dans votre branche courante. Par défaut, l'intégration est un merge ; avec --rebase, c'est un rebase.

Source : Pro Git book (CC BY-NC-SA 3.0), git-scm.com

    t_fetch["git fetch"] -->|"met a jour"| t_origin["origin/main (suivi)"]
    t_origin -->|"merge"| t_pullmerge["git pull (merge)"]
    t_origin -->|"rebase"| t_pullrebase["git pull --rebase (lineaire)"]
    t_pullmerge -->|"integre dans"| t_local["main locale"]
    t_pullrebase -->|"integre dans"| t_local
Critère git pull (merge) git pull --rebase
Effet fetch + merge fetch + rebase de vos commits locaux
Historique crée un commit de merge si divergence linéaire, pas de commit de merge
Lisibilité moyenne (merges « pull » parasites) élevée
Risque sûr réécrit vos commits locaux non poussés

🔹 Bonne pratique production : configurez git config --global pull.rebase true pour que git pull rebase par défaut. Cela évite les commits de merge « Merge branch 'main' of origin » qui polluent l'historique du dépôt-école. Le rebase ici ne réécrit que vos commits locaux non encore poussés — c'est sans danger.

🟨 Piège courant — git pull aveugle qui crée un merge surprise

Si vous avez des commits locaux et que le remote a avancé, un git pull (sans rebase) crée un commit de merge inattendu. Pire, sur une divergence, il peut ouvrir un conflit au pire moment. Réflexe : git fetch d'abord, regardez l'écart avec git status ou git log main..origin/main, puis intégrez en connaissance de cause.

3. push, push -u et l'upstream

git push envoie vos commits locaux vers le remote. La première fois, il faut indiquer où : git push -u origin <branche> pousse et établit le lien de suivi (upstream). Ensuite, un simple git push (et git pull) suffit, Git connaît la cible.

Source : Pro Git book (CC BY-NC-SA 3.0), git-scm.com

# Premier push d'une branche : pousse ET etablit l'upstream (-u)
git push -u origin main
# Ensuite, un simple push suffit
git push
# Pousse une branche de feature et l'associe a son homonyme distante
git push -u origin feature/navigation

4. force-with-lease vs force

Quand votre branche a été réécrite (rebase, amend) et que le remote a l'ancienne version, un git push normal est rejeté (non-fast-forward). On peut forcer :

  • git push --force : écrase la branche distante, sans vérifier. Si un collègue a poussé entre-temps, son travail est perdu.
  • git push --force-with-lease : force seulement si la branche distante est toujours à l'état que vous aviez au dernier fetch. Si quelqu'un a poussé depuis, le push est refusé. C'est le « force sûr ».
flowchart LR
    t_push["git push (rejete: non-fast-forward)"] -->|"option dangereuse"| t_force["--force : ecrase tout"]
    t_push -->|"option sure"| t_lease["--force-with-lease : verifie d'abord"]
    t_lease -->|"distant inchange"| t_ok["push accepte"]
    t_lease -->|"distant a evolue"| t_refus["push refuse (protege le collegue)"]

# Force SUR : echoue si le distant a evolue depuis votre dernier fetch
git push --force-with-lease
# Force BRUTAL : a eviter, ecrase le travail des autres sans verifier
git push --force

🟥 Anti-pattern — git push --force sur une branche partagée

Forcer brutalement sur main ou une branche de feature partagée détruit les commits que d'autres ont poussés entre-temps. C'est l'incident classique « j'ai écrasé le travail de Gabriel G. ». Utilisez toujours --force-with-lease, et idéalement protégez la branche côté forge (Jour 3) pour interdire tout force-push.

Partie TP intégré — Simuler un remote avec clone --bare

Vous allez créer un dépôt distant local (sans interface web) et y pousser le dépôt-école, puis simuler un second développeur qui récupère et collabore.

Prérequis du TP

  • Avoir terminé le TP1 : le dépôt tp1-depot-ecole avec ses commits existe.
  • Git 2.23+ et un terminal Bash.
  • Tout se passe en local : aucun accès réseau requis.

🔷 Note : un dépôt bare (--bare) est un dépôt sans working directory — il ne contient que le contenu de .git. C'est exactement ce qu'est un serveur Git : un point d'échange, pas un espace d'édition. On ne travaille jamais directement dedans.

Étape 1 : Créer le remote bare et l'associer

Objectif : transformer un dossier en serveur Git et y pousser le dépôt-école.

Explication : on crée un dépôt bare vide, on l'ajoute comme remote origin au dépôt-école, et on pousse main avec -u pour établir le suivi.

# Place-vous dans le repertoire parent du depot-ecole
cd ..
# Cree un depot BARE qui jouera le role de serveur (origin)
git init --bare depot-ecole.git

Initialized empty Git repository in /home/adrien/depot-ecole.git/

# Entre dans le depot-ecole du TP1
cd tp1-depot-ecole
# Ajoute le depot bare comme remote origin (chemin local en guise d'URL)
git remote add origin ../depot-ecole.git
# Pousse main et etablit l'upstream
git push -u origin main

Enumerating objects: 18, done.
...
To ../depot-ecole.git

 * [new branch]      main -> main
branch 'main' set up to track 'origin/main'.

Vérification d'état :

# Confirme le lien de suivi et l'absence d'ecart
git branch -vv

* main a1b2c3d [origin/main] docs: precise le mainteneur dans le README

🔵 Info : [origin/main] entre crochets confirme l'upstream. L'absence de ahead/behind signifie que local et distant sont synchronisés.

À vous !

  1. Créez le dépôt bare depot-ecole.git dans le répertoire parent.
  2. Depuis tp1-depot-ecole, ajoutez-le comme origin et poussez main avec -u.
  3. Vérifiez le suivi avec git branch -vv.

Erreurs fréquentes :

🟨 Symptôme Cause probable Comment le détecter Correction
remote origin already exists un origin est déjà défini git remote -v en liste un git remote remove origin puis ré-ajouter
src refspec main does not match any aucun commit sur main TP1 non terminé refaire le TP1 (au moins un commit)
push accepté mais pas de suivi -u oublié git branch -vv n'affiche pas [origin/main] git push -u origin main

Étape 2 : Cloner (simuler un second développeur)

Objectif : créer deux clones du remote, comme si Adrien et Guive travaillaient chacun sur sa machine.

# Reviens dans le repertoire parent
cd ..
# Clone d'Adrien (relie automatiquement a origin)
git clone depot-ecole.git clone-adrien
# Clone de Guive
git clone depot-ecole.git clone-guive

Vérification d'état :

# Le clone connait deja origin et le suivi
cd clone-adrien && git branch -vv && git log --oneline -1

* main a1b2c3d [origin/main] docs: precise le mainteneur dans le README

🔵 Info : git clone configure automatiquement origin et l'upstream — pas besoin de git remote add ni de -u sur un dépôt cloné.

À vous !

Créez les deux clones clone-adrien et clone-guive. Vérifiez dans chacun que origin est configuré (git remote -v) et que l'historique est complet (git log --oneline).

Étape 3 : Push depuis un clone, fetch dans l'autre

Objectif : matérialiser le flux complet. Adrien pousse un commit, Guive le récupère avec fetch puis intègre.

# --- Cote Adrien : produit un commit et le pousse ---
cd ../clone-adrien
# Ajoute une ligne de contact dans le README
printf '\nContact : adrien.vossough\n' >> README.md
git add README.md
git commit -m "docs: ajoute le contact mainteneur"
git push

To /home/adrien/depot-ecole.git
   a1b2c3d..e4f5g6h  main -> main

# --- Cote Guive : recupere SANS integrer, inspecte, puis integre ---
cd ../clone-guive
# fetch : met a jour origin/main, ne touche pas a la branche locale
git fetch
# Constate l'ecart : la branche locale est "behind 1"
git status

On branch main
Your branch is behind 'origin/main' by 1 commit, and can be fast-forwarded.
  (use "git pull" to update your local branch)

# Inspecte ce qui arrive AVANT d'integrer
git log --oneline main..origin/main
# Integre proprement (rebase pour rester lineaire)
git pull --rebase

Vérification d'état :

# Guive est maintenant a jour : meme dernier commit qu'Adrien
git log --oneline -1

e4f5g6h docs: ajoute le contact mainteneur

À vous !

  1. Depuis clone-adrien, faites un commit (au choix, ex. une ligne dans document.txt) et git push.
  2. Depuis clone-guive, faites git fetch, observez le behind dans git status, listez l'écart avec git log --oneline main..origin/main.
  3. Intégrez avec git pull --rebase. Vérifiez que les deux clones ont le même dernier commit.

Erreurs fréquentes :

🟨 Symptôme Cause probable Comment le détecter Correction
Updates were rejected (non-fast-forward) les deux clones ont divergé message au push git pull --rebase puis git push
git status ne montre pas le behind git fetch oublié origin/main pas à jour lancer git fetch d'abord
Commit de merge « Merge branch main » inattendu git pull sans --rebase git lg montre un merge configurer pull.rebase true ; ici, refaire avec --rebase

Étape 4 : Provoquer et résoudre un non-fast-forward avec force-with-lease

Objectif : comprendre le rejet non-fast-forward et la différence --force / --force-with-lease, sans casser le travail des autres.

Contexte : Adrien réécrit son dernier commit (amend) après l'avoir poussé. Le remote a l'ancienne version : le push normal est rejeté.

cd ../clone-adrien
# Reecrit le dernier commit (change le message) -> nouveau hash
git commit --amend -m "docs: ajoute le contact du mainteneur (reformule)"
# Le push normal est REJETE (l'historique a diverge)
git push

 ! [rejected]        main -> main (non-fast-forward)
error: failed to push some refs to '/home/adrien/depot-ecole.git'

# Comme c'est VOTRE reecriture et que personne d'autre n'a pousse, force SUR :
git push --force-with-lease

 + e4f5g6h...h7i8j9k main -> main (forced update)

🟨 Warning : --force-with-lease réussit ici parce que personne d'autre n'a poussé depuis votre dernier fetch. Si Guive avait poussé entre-temps, il aurait échoué et protégé son travail — c'est tout l'intérêt par rapport à --force.

À vous !

  1. Depuis clone-adrien, amendez le dernier commit (git commit --amend).
  2. Tentez git push : observez le rejet non-fast-forward.
  3. Poussez avec git push --force-with-lease.
  4. Depuis clone-guive, faites git fetch puis git log --oneline et git status pour constater la divergence côté Guive (elle devra rebaser pour se réaligner — anticipe le Jour 2).

Erreurs fréquentes :

🟨 Symptôme Cause probable Comment le détecter Correction
--force-with-lease rejeté un fetch a rafraîchi le lease, ou un collègue a poussé message stale info git fetch, inspecter, puis re-décider (ne jamais --force à l'aveugle)
Guive coincée après le force d'Adrien historique réécrit côté distant git status diverge git pull --rebase côté Guive (vu au Jour 2)
--force tapé par réflexe mauvaise habitude aucun garde-fou adopter systématiquement --force-with-lease

Récap

🟢 Ce que vous savez maintenant faire :

  • Ajouter un remote, pousser avec git push -u et lire le suivi via git branch -vv.
  • Distinguer git fetch (récupérer) de git pull / git pull --rebase (intégrer).
  • Simuler un serveur Git avec git clone --bare et échanger entre deux clones.
  • Diagnostiquer un rejet non-fast-forward et le résoudre avec --force-with-lease.

🟨 Pièges à retenir :

  • git pull sans --rebase crée des commits de merge parasites en cas de divergence.
  • git push --force écrase le travail des autres ; --force-with-lease le protège.
  • origin/main n'est pas le serveur en direct, mais votre dernière photo de lui (mise à jour au fetch).

🔷 Points clés à retenir :

  • Un dépôt bare n'a pas de working directory : c'est un point d'échange, jamais un espace d'édition.
  • Le couple fetch + git log main..origin/main permet d'inspecter avant d'intégrer.
  • git clone configure origin et l'upstream automatiquement.
