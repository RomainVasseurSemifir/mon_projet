Module 2.1 — Merge vs Rebase

À la fin de ce module, vous saurez : - Distinguer une fusion fast-forward d'une fusion avec commit de merge et forcer ce dernier avec --no-ff. - Décrire en une phrase ce que produit un rebase sur le DAG et en quoi il diffère d'un merge. - Choisir merge ou rebase selon le contexte à l'aide d'un arbre de décision. - Réécrire une série de commits avec git rebase -i (pick, reword, edit, squash, fixup, drop). - Énoncer et appliquer la règle critique « ne jamais rebaser une branche partagée déjà poussée ».

Au Jour 1, le dépôt-école de l'équipe (Adrien Vossough, Guive Voss, François S.) a un historique linéaire de document.txt poussé sur origin. Aujourd'hui la branche diverge : Guive ouvre feature/amelioration pendant qu'Adrien continue sur main. Deux historiques avancent en parallèle — il faut les intégrer. Git offre deux opérations pour cela : merge et rebase.

1. Le point de départ : deux branches qui divergent

Avant d'intégrer, comprenons l'état de départ. Une branche est un simple pointeur mobile vers un commit. Quand deux branches reçoivent chacune un commit après leur dernier ancêtre commun, l'historique diverge : il n'y a plus de ligne droite entre elles.

Source : Pro Git book (CC BY-NC-SA 3.0), git-scm.com

gitGraph
    commit id: "C0"
    commit id: "C1-base"
    branch feature
    checkout main
    commit id: "C2-main"
    checkout feature
    commit id: "C3-feat"

Sur ce graphe, main et feature partent du même ancêtre C1-base, puis chacune ajoute son commit. Pour ramener le travail de feature dans main, deux stratégies s'offrent à vous.

2. Le merge : préserver l'historique réel

2.1 Fast-forward (avance rapide)

Le fast-forward se produit quand la branche cible (main) n'a aucun commit depuis le point de départ de feature. Git n'a alors rien à fusionner : il se contente d'avancer le pointeur main jusqu'au dernier commit de feature. Aucun commit de merge n'est créé, l'historique reste une ligne droite.

git switch main          # se placer sur la branche d'accueil
git merge feature/x      # fast-forward si main n'a pas bouge

2.2 Commit de merge (--no-ff)

Quand les deux branches ont avancé (cas de la divergence ci-dessus), un fast-forward est impossible. Git crée un commit de merge : un commit spécial à deux parents qui réconcilie les deux historiques. Ce commit matérialise pour toujours « ici, telle branche a été intégrée ».

Source : Pro Git book (CC BY-NC-SA 3.0), git-scm.com

L'option --no-ff (no fast-forward) force la création de ce commit de merge même quand un fast-forward serait possible. Intérêt : tracer explicitement le périmètre d'une fonctionnalité.

git switch main
git merge --no-ff feature/amelioration -m "merge: integration amelioration"
# -> cree un commit de merge meme si un fast-forward etait possible

gitGraph
    commit id: "M0"
    commit id: "M1-base"
    branch feature
    checkout main
    commit id: "M2-main"
    checkout feature
    commit id: "M3-feat"
    checkout main
    merge feature id: "M4-merge"

🔹 Bonne pratique production : sur le dépôt-école, l'équipe d'Adrien utilise --no-ff pour les fusions de fonctionnalités afin de garder visible le « bloc » de chaque feature. La logique est reprise côté forge par l'option Create a merge commit (Module 3.1).

3. Le rebase : réécrire pour linéariser

Le rebase ne fusionne pas : il rejoue les commits de votre branche, un par un, par-dessus la pointe d'une autre branche. Le résultat est un historique linéaire, comme si vous aviez développé à partir du dernier état de main.

Source : Pro Git book (CC BY-NC-SA 3.0), git-scm.com

git switch feature/amelioration
git rebase main          # rejoue les commits de feature au-dessus de main

Après le rebase, les commits de feature ont de nouveaux SHA (ce sont des copies, pas les originaux) et pointent désormais sur la pointe de main. L'ancêtre commun a, en pratique, été déplacé.

Source : Pro Git book (CC BY-NC-SA 3.0), git-scm.com

Il reste ensuite à faire avancer main : comme feature descend maintenant en ligne droite de main, un simple merge se résout en fast-forward.

Source : Pro Git book (CC BY-NC-SA 3.0), git-scm.com

git switch main
git merge feature/amelioration   # fast-forward : historique 100% lineaire

En cas de problème pendant un rebase (conflit), trois commandes pilotent la suite :

git rebase --continue    # poursuivre apres avoir resolu un conflit
git rebase --skip        # ignorer le commit en cours de rejeu
git rebase --abort       # tout annuler, revenir a l'etat d'avant rebase

4. Le rebase interactif : nettoyer une série de commits

git rebase -i (interactif) ouvre un éditeur listant les commits à rejouer. Vous choisissez une action pour chacun. C'est l'outil pour transformer un historique brouillon (« wip », « oups », « fix typo ») en une série propre avant de proposer une revue.

git rebase -i HEAD~5     # editer les 5 derniers commits

Action Effet Cas d'usage type sur le dépôt-école
pick garde le commit inchangé conserver un commit déjà propre
reword rejoue mais ouvre l'éditeur du message corriger « amelioration » en feat: enrichir document.txt
edit s'arrête sur ce commit pour l'amender scinder un commit trop gros de François

🔹 Astuce : git commit --fixup <hash> puis git rebase -i --autosquash placent et marquent automatiquement les correctifs en fixup. Gain de temps lors du nettoyage avant revue.

6. La règle d'or des branches partagées

🟥 Règle critique : ne jamais rebaser une branche déjà poussée et partagée. Le rebase recrée des commits avec de nouveaux SHA. Si un collègue (Luigi M.) a déjà basé son travail sur les anciens commits, votre push --force réécrit l'historique distant : Luigi se retrouve avec des doublons et des conflits inextricables au prochain pull.

🟨 Warning — Le piège du « petit force pour nettoyer » Rebaser une branche de fonctionnalité personnelle non encore partagée est sain et recommandé avant la revue. Le danger n'apparaît qu'à partir du moment où quelqu'un d'autre a tiré ces commits. En cas de doute, merge. Si vous devez réécrire une branche partagée, c'est une décision d'équipe annoncée, jamais un réflexe individuel.

🟨 Warning — git pull qui « fait un merge » non désiré Par défaut, git pull fait un fetch + merge, ce qui crée des commits de merge parasites sur les branches de suivi. Pour un historique propre, configurez git config --global pull.rebase true (déjà recommandé dans la config du Jour 1).

Ce qu'il faut retenir

  1. Fast-forward = avance de pointeur sans commit de merge ; possible seulement si la cible n'a pas bougé.
  2. merge --no-ff = crée toujours un commit de merge à deux parents, qui trace le périmètre de la feature.
  3. rebase = rejoue vos commits ailleurs, leur donne de nouveaux SHA, et produit un historique linéaire.
  4. rebase -i = nettoie une série de commits (reword/squash/fixup/drop) avant la revue.
  5. 🟥 Jamais de rebase sur une branche partagée poussée. En partagé : merge.
